# ddsl

`ddsl` is an intent-driven container build DSL compiler packaged as a Maven
plugin and a command-line tool.

Instead of hand-authoring Dockerfile mechanics, you describe the build you want:
base image metadata, language-aware build steps, tools, artifacts, and runtime
behavior. The compiler validates that intent and emits a deterministic
Dockerfile.

This is not a Dockerfile wrapper and not a template language. Dockerfile is the
first backend target, not the authoring surface.

The active implementation is Java and Maven-native:

- `ddsl-core`: reusable compiler library
- `ddsl-cli`: command-line frontend for local testing and CI
- `ddsl-maven-plugin`: Maven goal for project builds

## Why This Exists

Dockerfiles mix intent with low-level mechanics:

- package-manager commands,
- mutable versus immutable image behavior,
- shell and entrypoint details,
- build artifact paths,
- runtime copy rules,
- language-specific build incantations.

`ddsl` moves those concerns into a compiler. The DSL stays focused on what the
container should contain and how it should run; the compiler owns how that maps
to Docker.

## Quickstart

Build and test the Java Maven reactor:

```sh
mvn package
```

Run the command-line transpiler on Linux or macOS:

```sh
./scripts/ddsl examples/java.dsl
./scripts/ddsl examples/java.dsl -o target/Dockerfile
```

Run the command-line transpiler on Windows:

```bat
scripts\ddsl.cmd examples\java.dsl
scripts\ddsl.cmd examples\java.dsl -o target\Dockerfile
```

The scripts build `ddsl-cli` automatically if the executable jar is missing.
You can also run the jar directly after `mvn package`:

```sh
java -jar ddsl-cli/target/ddsl-cli-0.1.0-SNAPSHOT-all.jar examples/java.dsl
```

## Maven Plugin

Use the Maven plugin from a project by adding `ddsl-maven-plugin` to the
project `pom.xml`:

```xml
<plugin>
  <groupId>city.fini</groupId>
  <artifactId>ddsl-maven-plugin</artifactId>
  <version>0.1.0-SNAPSHOT</version>
  <configuration>
    <input>${project.basedir}/container.dsl</input>
    <output>${project.build.directory}/Dockerfile</output>
  </configuration>
  <executions>
    <execution>
      <goals>
        <goal>transpile</goal>
      </goals>
      <phase>generate-resources</phase>
    </execution>
  </executions>
</plugin>
```

Run the goal directly:

```sh
mvn city.fini:ddsl-maven-plugin:0.1.0-SNAPSHOT:transpile \
  -Dddsl.input=container.dsl \
  -Dddsl.output=target/Dockerfile
```

Or bind it to `generate-resources` as shown above so it runs during the normal
Maven lifecycle.

## DSL At A Glance

```text
let REGISTRY = "docker.io"

stage build-java as build {
  base chainguard {
    variant dev
    distro  wolfi
    image   "{{REGISTRY}}/base-images/chainguard/openjdk:24.0.1-dev"
  }

  workdir "/app"
  user "root"

  copy ["mvnw", "pom.xml"] to "./"

  tool maven {
    goals ["dependency:go-offline"]
  }

  copy ["src", "spec"] to "./"

  tool maven {
    goals     ["package"]
    skipTests true
  }

  produces artifact "application" at "/app/target/application-*.jar"
}

stage image as image {
  base chainguard {
    variant runtime
    distro  wolfi
    image   "{{REGISTRY}}/base-images/chainguard/openjdk:24.0.1"
  }

  workdir "/app"
  user "nonroot"

  copy artifact "application" as "app.jar" to "/app/app.jar"

  runtime java {
    jar "/app/app.jar"
  }

  expose 8000
}
```

Generated shape:

```dockerfile
# Generated by ddsl. Do not edit by hand.

FROM docker.io/base-images/chainguard/openjdk:24.0.1-dev AS build-java
WORKDIR /app
USER root
COPY mvnw pom.xml ./
RUN mvn -B dependency:go-offline
COPY src spec ./
RUN mvn -B package -DskipTests
# produces artifact application at /app/target/application-*.jar

FROM docker.io/base-images/chainguard/openjdk:24.0.1 AS image
WORKDIR /app
USER nonroot
# copy artifact application as target artifact app.jar from build-java:/app/target/application-*.jar to /app/app.jar
COPY --from=build-java /app/target/application-*.jar /app/app.jar
ENTRYPOINT ["java","-jar","/app/app.jar"]
EXPOSE 8000
```

## Artifact Model

Artifacts have labels. Labels are stable names inside the DSL; paths describe
where the artifact exists in the build stage.

```text
produces artifact "application" at "/app/target/application-*.jar"
```

That says the build stage produces an artifact labeled `application`. The path
may be exact or may use a wildcard for versioned outputs, such as Maven jars.

Runtime stages copy by label and can give the copied artifact a target name:

```text
copy artifact "application" as "app.jar" to "/app/app.jar"
```

That says: copy the build artifact labeled `application`, treat it as target
artifact `app.jar`, and place it at `/app/app.jar`.

For compiler-generated outputs, exactness still matters. For example,
`tool go { build true }` uses the artifact path as the `go build -o` target, so
that artifact path must be exact rather than wildcarded.

## Supported Constructs

- Variables: `let NAME = "value"`
- String interpolation: `"{{REGISTRY}}/image"`
- Stages: `stage <name> as build|test|image { ... }`
- Bases: `base chainguard { variant ... distro ... image ... }`
- Stage configuration: `workdir`, `user`
- Mutating operations: `update`, `tools { build [...] }`
- Source copies: `copy ["src", "spec"] to "./"`
- Artifact copies: `copy artifact "application" as "app.jar" to "/app/app.jar"`
- Language-aware tools: Go, Maven, Python, Node
- Artifacts: `produces artifact "name" at "/path/or/pattern"`
- Runtime blocks: binary, Java, Python, Node
- Port exposure: `expose 8000`
- Carried tools: `carry tool aws-cli`

## Validation

The compiler validates before generating Dockerfiles. It rejects:

- image stages that are not last,
- `update` in image stages,
- `tools` blocks in image stages,
- runtime blocks outside image stages,
- artifact copies that reference missing artifacts,
- duplicate stage names,
- duplicate artifact names,
- carried tools that were not declared in a prior build/test tools block,
- package-manager mutation in immutable runtime images,
- unresolved interpolation variables,
- unsupported base metadata combinations,
- Go build output artifacts that use wildcard paths.

Diagnostics are structured and include spans where practical.

## Compiler-Owned Metadata

The compiler does not infer behavior from image names such as `-dev`, `-slim`,
or `runtime`. It uses explicit DSL fields plus compiler-owned metadata.

Initial package-manager metadata:

| Distro | Package manager |
| --- | --- |
| `wolfi` | `apk` |
| `alpine` | `apk` |
| `debian` | `apt` |
| `ubuntu` | `apt` |
| `rhel` | `dnf` |
| `distroless` | none |
| `windows` | unsupported |

Initial mutability metadata:

| Variant | Mutable |
| --- | --- |
| `dev` | yes |
| `runtime` | no |

Initial shell availability:

| Distro | Shell |
| --- | --- |
| `wolfi`, `alpine`, `debian`, `ubuntu`, `rhel` | yes |
| `distroless`, `windows` | no |

## Language Support

Go:

```text
tool go {
  build true
}
```

Maven:

```text
tool maven {
  goals ["package"]
  skipTests true
}
```

Python:

```text

    requirements "requirements.txt"
  }
}
```

Node:

```text
tool node {
  install prod
}
```

## Project Layout

```text
ddsl-core/
  src/main/java/city/fini/ddsl/
    ast/                    typed parsed AST
    diagnostics/            structured diagnostics
    generator/              Dockerfile backend
    metadata/               distro, mutability, package-manager, tool metadata
    model/                  semantic model
    parser/                 lexer and recursive-descent parser
    semantics/              variable resolution, model construction, validation
    DdslCompiler.java       public compiler API

ddsl-cli/
  src/main/java/...         CLI frontend

ddsl-maven-plugin/
  src/main/java/...         Maven `ddsl:transpile` goal

examples/                  DSL examples
expected/                  golden Dockerfile outputs
```

## Pipeline

1. Parse source with `pest`.
2. Build a typed AST.
3. Resolve variables and string interpolation.
4. Build a semantic model.
5. Validate stage ordering, artifact references, mutability, tools, runtime rules,
   and metadata support.
6. Generate a deterministic Dockerfile.

## Current Limitations

- Dockerfile is the only backend.
- Windows bases are rejected by the initial backend.
- Distroless is modeled as having no package manager and no shell.
- Carried-tool filesystem mappings are intentionally conservative.
- Carried paths currently exist for `git`, `ca-certificates`, and `aws-cli`.
- Wildcard artifact paths are supported for copying build outputs, but the
  compiler does not inspect the build context to prove how many files match.
